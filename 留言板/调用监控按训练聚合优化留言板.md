# AI调用监控按训练聚合优化留言板

> 目标：将教师端“调用监控”中的“最近调用日志”从“每次患者问答一条日志”优化为“每次训练一条患者对话聚合日志”。  
> 面向后续接手的 AI 或开发者。核心原则是：保留底层原始 LLM 调用日志，优先在查询层或展示层聚合，不影响训练速度、评分、学生端体验和已有运维统计。

## 一、问题背景

教师端进入“调用监控”后，在“最近调用日志”中会看到大量“患者对话”记录。当前系统的虚拟患者对话是每次学生提问都调用一次大模型，因此一次训练如果有 15 轮问答，就会产生约 15 条 `patient_chat` 日志。

当班级训练次数增加后，会出现以下问题：

1. “最近调用日志”分页数量快速膨胀。
2. 教师想看“一次训练消耗多少 Token / 多少钱 / 平均延迟多少”时，需要手动把很多条问答日志加总。
3. 页面上显示的“患者对话”更像技术调用明细，不符合教师按训练记录理解成本和性能的习惯。
4. 训练越多，排查单次训练成本和稳定性越不直观。

本优化只针对教师端最近日志的展示粒度：

- 原来：每次问答一条 `patient_chat`。
- 目标：每次训练一条“训练对话”聚合记录。

## 二、当前代码链路定位

后续修改前先检查以下文件。

### 1. LLM 日志写入

位置：

- `backend/services/llm_service.py`
- `backend/services/llm_logging.py`
- `backend/models.py`

当前链路：

1. `call_llm_stream(... purpose="patient_chat", record_id=..., case_id=...)` 每次患者回复都会记录一次日志。
2. `_log_llm_success()` / `_log_llm_failure()` 会异步调用 `services.llm_logging.log_llm_call()`。
3. `log_llm_call()` 写入 `llm_call_logs` 表。
4. `LLMCallLog.record_id` 指向一次训练记录，因此同一训练下的所有患者问答日志可以通过 `record_id` 聚合。

重要字段：

- `record_id`：训练记录 ID，是聚合依据。
- `purpose`：`patient_chat` 表示虚拟患者对话。
- `latency_ms`：单次问答延迟。
- `prompt_tokens`、`completion_tokens`、`total_tokens`：单次问答 Token。
- `estimated_cost`：单次问答费用。
- `status`：单次调用状态。
- `created_at`：单次调用时间。

### 2. 教师端日志接口

位置：

- `backend/routers/admin.py`
- `backend/schemas.py`

当前接口：

```text
GET /api/admin/llm-logs
```

当前行为：

1. 直接查询 `LLMCallLog`。
2. 按 `created_at desc` 排序。
3. 按原始调用分页。
4. 返回 `LLMLogListResponse(items, total, page, page_size)`。

因此，只要 `patient_chat` 调用多，分页就会变多。

### 3. 前端展示

位置：

- `frontend/src/components/teacher/MonitorTab.jsx`
- `frontend/src/api.js`

当前行为：

1. `getLLMLogs(params)` 调用 `/api/admin/llm-logs`。
2. `MonitorTab.jsx` 在“最近调用日志”表格逐条展示日志。
3. 表格列为：时间 / 用途 / 状态 / 延迟 / Token / 费用。
4. `patient_chat` 显示为“患者对话”，但本质是一轮问答，而不是一次训练。

## 三、推荐总体方案

优先采用：

> 保留原始 `llm_call_logs`，只在教师端日志查询/展示时聚合同一训练下的 `patient_chat`。

不要采用：

1. 不要停止写入每轮 `patient_chat` 原始日志。
2. 不要删除历史原始日志。
3. 不要在训练过程中维护额外聚合表作为第一版方案。
4. 不要改变 LLM 调用流程。
5. 不要新增任何 LLM 调用。

原因：

1. 原始日志对排查慢请求、失败请求、模型波动仍然有价值。
2. 查询层聚合不影响学生训练速度。
3. 不需要改数据库表结构即可完成第一版。
4. 对评分、问答、病例、训练记录等功能影响最小。

## 四、后端修改建议

### 1. 聚合口径

只聚合：

```text
purpose = "patient_chat"
record_id is not null
```

不聚合：

```text
purpose = "scoring"
purpose = "qa"
purpose = "summary"
purpose = "other"
record_id is null 的 patient_chat 异常日志
```

聚合单位：

```text
同一个 record_id 下的全部 patient_chat 调用 = 一条训练对话聚合记录
```

### 2. 聚合字段规则

建议聚合结果字段如下：

| 字段 | 规则 |
|------|------|
| `id` | 可使用 `record_id` 或 `max(LLMCallLog.id)`，前端 key 必须稳定 |
| `record_id` | 分组字段 |
| `case_id` | `max(case_id)` 或从训练记录关联病例 |
| `user_id` | `max(user_id)` 或从训练记录关联学生 |
| `purpose` | 固定为 `patient_chat` |
| `purpose_label` | 建议前端显示“训练对话” |
| `call_count` | `count(*)`，表示本次训练内患者对话轮数 |
| `latency_ms` | `avg(latency_ms)`，表示整次训练平均延迟 |
| `avg_latency_ms` | 可选，和 `latency_ms` 同值，便于语义清晰 |
| `prompt_tokens` | `sum(prompt_tokens)` |
| `completion_tokens` | `sum(completion_tokens)` |
| `total_tokens` | `sum(total_tokens)` |
| `estimated_cost` | `sum(estimated_cost)` |
| `token_estimated` | 如果任意子日志是估算，则聚合结果为 `1`；否则为 `0` |
| `status` | 如果任意子日志非 `success`，显示 `failed` 或 `partial_failed`；否则 `success` |
| `error_count` | `sum(status != "success")` |
| `created_at` | `max(created_at)`，表示最近一次对话时间，用于排序 |
| `first_called_at` | `min(created_at)` |
| `last_called_at` | `max(created_at)` |
| `student_name` | 可选，从 `TrainingRecord.user` 或 `User.display_name` 获取 |
| `case_name` | 可选，从 `TrainingRecord.case` 或 `Case.name` 获取 |

### 3. 接口选择

推荐有两个可选实现，优先级如下。

#### 方案 A：保留 `/api/admin/llm-logs`，增加聚合参数

推荐参数：

```text
aggregate_patient_chat=true
```

默认值建议：

```text
true
```

理由：

1. 教师端默认就应该看聚合后的训练级日志。
2. 老接口路径不变，前端改动少。
3. 未来若需要查看原始调用明细，可传 `aggregate_patient_chat=false`。

注意：

如果担心影响已有调用方，可让前端显式传 `aggregate_patient_chat=true`，后端默认保持旧行为。这取决于后续开发者对兼容性的判断。

#### 方案 B：新增 `/api/admin/llm-training-logs`

用途：

```text
GET /api/admin/llm-training-logs
```

只返回聚合后的训练级日志。

优点：

1. 不破坏旧接口。
2. 原始调用日志与训练聚合日志语义更清晰。

缺点：

1. 需要新增前端 API 方法。
2. 前端 MonitorTab 要切换接口。

### 4. 查询实现要点

在 SQLite + SQLAlchemy 下，可以先用分组查询实现。

核心思路：

```python
patient_chat_groups = db.query(
    LLMCallLog.record_id.label("record_id"),
    func.max(LLMCallLog.id).label("id"),
    func.max(LLMCallLog.user_id).label("user_id"),
    func.max(LLMCallLog.case_id).label("case_id"),
    func.count().label("call_count"),
    func.avg(LLMCallLog.latency_ms).label("latency_ms"),
    func.sum(LLMCallLog.prompt_tokens).label("prompt_tokens"),
    func.sum(LLMCallLog.completion_tokens).label("completion_tokens"),
    func.sum(LLMCallLog.total_tokens).label("total_tokens"),
    func.max(LLMCallLog.token_estimated).label("token_estimated"),
    func.sum(LLMCallLog.estimated_cost).label("estimated_cost"),
    func.sum(case((LLMCallLog.status != "success", 1), else_=0)).label("error_count"),
    func.min(LLMCallLog.created_at).label("first_called_at"),
    func.max(LLMCallLog.created_at).label("created_at"),
).filter(
    LLMCallLog.purpose == "patient_chat",
    LLMCallLog.record_id.isnot(None),
)
```

实际实现时要把现有筛选条件同步进去：

1. `status`
2. `date_from`
3. `date_to`
4. `purpose`
5. `page`
6. `page_size`

筛选原则：

1. 如果筛选 `purpose=patient_chat`，只返回训练聚合日志。
2. 如果筛选 `purpose=scoring`，仍返回原始评分调用。
3. 如果不筛选 `purpose`，建议返回：
   - 聚合后的 `patient_chat` 训练日志。
   - 原始的 `scoring` / `qa` / `summary` / `other` 日志。
4. 如果筛选 `status=failed`，聚合日志应返回有失败子调用的训练。
5. 如果筛选 `status=success`，聚合日志应返回全部子调用均成功的训练。

分页原则：

1. 先形成最终展示列表。
2. 再按 `created_at desc` 排序。
3. 再分页。

如果 SQL union 实现复杂，第一版可以：

1. 用 SQL 聚合取 `patient_chat`。
2. 用 SQL 查询非 `patient_chat` 原始日志。
3. 在 Python 中合并、排序、分页。

注意：

如果数据量非常大，Python 合并分页可能不适合长期生产。但当前教学试运行阶段可以接受；后续上 PostgreSQL 时再改成 SQL union/subquery。

## 五、Schema 修改建议

位置：

- `backend/schemas.py`

现有：

```python
class LLMCallLogItem(BaseModel):
    ...
```

建议增加可选字段，兼容原始日志和聚合日志：

```python
call_count: int = 1
avg_latency_ms: Optional[int] = None
error_count: int = 0
first_called_at: Optional[datetime] = None
last_called_at: Optional[datetime] = None
student_name: Optional[str] = None
case_name: Optional[str] = None
is_aggregated: bool = False
```

兼容策略：

1. 原始调用日志：`call_count=1`，`is_aggregated=false`。
2. 训练聚合日志：`call_count=本次训练 patient_chat 条数`，`is_aggregated=true`。
3. 保留 `latency_ms` 字段，但聚合日志中它表示平均延迟。
4. 前端显示时可以优先使用 `avg_latency_ms ?? latency_ms`。

不要删除旧字段：

- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `estimated_cost`
- `created_at`
- `status`

## 六、前端修改建议

位置：

- `frontend/src/api.js`
- `frontend/src/components/teacher/MonitorTab.jsx`

### 1. API 调用

如果采用方案 A：

```js
getLLMLogs({ ...params, aggregate_patient_chat: true })
```

如果采用方案 B：

新增：

```js
export function getLLMTrainingLogs(params = {}) {
  return api.get("/admin/llm-training-logs", { params });
}
```

### 2. 表格文案

将标题：

```text
最近调用日志
```

改为：

```text
最近训练调用日志
```

用途显示规则：

1. 聚合后的 `patient_chat` 显示为：

```text
训练对话（N轮）
```

2. 普通 `scoring` 显示为：

```text
评分
```

3. 普通 `qa` 显示为：

```text
问答
```

### 3. 表格列建议

建议列：

```text
时间 / 用途 / 训练 / 状态 / 平均延迟 / Token总量 / 总费用
```

如果空间不足，可以保持原列：

```text
时间 / 用途 / 状态 / 延迟 / Token / 费用
```

但含义要在代码或 tooltip 中明确：

1. 聚合患者对话行的“延迟”是整次训练平均延迟。
2. 聚合患者对话行的“Token”是整次训练总 Token。
3. 聚合患者对话行的“费用”是整次训练总费用。

### 4. 分页

分页仍使用后端返回的：

```text
total / page / page_size
```

期望变化：

如果某学生一次训练有 20 轮患者对话：

- 修改前：20 条患者对话日志。
- 修改后：1 条训练对话日志，显示 `训练对话（20轮）`。

## 七、统计卡片是否调整

本次需求只明确“最近调用日志”按训练聚合，不要求顶部统计卡片改变。

默认建议：

1. 今日调用次数、成功率、平均延迟、预估费用继续按原始 LLM 调用统计。
2. 按用途统计继续按原始调用统计。
3. 近 30 天每日调用趋势继续按原始调用统计。

理由：

这些指标反映的是底层 LLM 服务压力和调用稳定性，保留原始调用口径更适合运维。

只有当后续产品明确要求“所有监控都按训练聚合”时，才同步调整统计卡片和趋势图。

## 八、测试方案

### 1. 后端聚合测试

建议新增或扩展后端测试：

1. 构造同一个 `record_id` 下 10 条 `purpose="patient_chat"` 日志。
2. 构造 1 条 `purpose="scoring"` 日志。
3. 调用聚合后的日志接口。
4. 断言返回：
   - patient_chat 聚合为 1 条。
   - `call_count == 10`。
   - `latency_ms` 等于 10 条延迟平均值。
   - `total_tokens` 等于 10 条总和。
   - `estimated_cost` 等于 10 条总和。
   - scoring 仍为 1 条原始日志。

失败状态测试：

1. 同一 `record_id` 下 9 条成功、1 条失败。
2. 聚合行应显示 `status="failed"` 或 `status="partial_failed"`。
3. `error_count == 1`。

筛选测试：

1. `purpose=patient_chat` 只返回训练聚合记录。
2. `purpose=scoring` 只返回评分原始记录。
3. `status=success` 不返回含失败子调用的聚合记录。
4. `status=failed` 返回含失败子调用的聚合记录。

### 2. 前端测试

如果已有前端测试体系，建议补充：

1. mock `/api/admin/llm-logs` 返回 `is_aggregated=true, call_count=10`。
2. 验证页面显示“训练对话（10轮）”。
3. 验证 Token 和费用显示总量。
4. 验证分页总数使用后端返回的聚合后 `total`。

### 3. 人工验收

使用教师账号进入：

```text
管理后台 -> 调用监控 -> 最近训练调用日志
```

验收点：

1. 同一次训练不再出现十几条“患者对话”。
2. 每次训练只显示一条“训练对话（N轮）”。
3. 平均延迟符合各轮患者对话平均值。
4. Token 和费用符合整次训练总和。
5. 评分、问答仍按单次调用显示。
6. 学生训练流程无变化。
7. 教师查看训练记录、评分复核、病例管理无变化。

## 十、不得影响的内容

后续实现必须避免影响以下模块：

1. 学生端训练对话速度。
2. LLM 请求参数和重试机制。
3. 自动评分流程。
4. 护理问答流程。
5. 训练记录保存和导出。
6. 教师复核。
7. 病例管理。
8. 原始 `llm_call_logs` 写入和保留。

特别注意：

不要为了减少分页而不写入每轮原始日志。原始日志仍然是排查慢调用、失败调用和成本异常的基础数据。

## 十一、推荐最终效果

修改完成后，教师端最近日志应呈现类似：

| 时间 | 用途 | 训练 | 状态 | 平均延迟 | Token总量 | 总费用 |
|------|------|------|------|----------|-----------|--------|
| 2026-05-28 14:30 | 训练对话（12轮） | 病例1 / 学生A | 成功 | 1380ms | 18240 | ¥0.0123 |
| 2026-05-28 14:28 | 评分 | 病例1 / 学生A | 成功 | 5200ms | 3260 | ¥0.0048 |
| 2026-05-28 14:25 | 问答 | - | 成功 | 2100ms | 920 | ¥0.0011 |

这样教师看到的是“训练级成本和性能”，分页数量按训练次数下降，同时底层运维仍保留每轮原始调用明细。
