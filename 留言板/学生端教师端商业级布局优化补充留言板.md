# AI学生端教师端商业级布局优化留言板（v1.15 截图问题补充）

> 编写日期：2026-05-27  
> 编写目的：根据当前系统源码、文档和用户提供的两张粗糙页面截图，记录后续 AI 或开发者应如何把学生端、教师端网页系统优化到商业可交付水平。  
> 重要说明：本留言板只记录分析、步骤和验收标准。本次未修改前端源码、后端源码、数据库或构建产物。

## 0. 给下一个 AI 的第一句话

请不要一上来重写页面，也不要只调颜色。当前系统已经有设计系统雏形，但截图显示所有页面像浏览器默认样式，最优先要确认并修复“全局 CSS 没有进入应用”的问题。只有样式入口恢复后，才能判断现有 v1.14 界面打磨到底完成到什么程度，再继续做真正的商业级布局优化。

当前最关键发现：

- `frontend/src/styles/tokens.css` 已存在，包含完整颜色、字体、间距、圆角、阴影等设计令牌。
- `frontend/src/styles/index.css` 已存在，且第一行通过 `@import './tokens.css';` 引入 token。
- 但 `frontend/src/main.jsx` 当前只渲染 `<App />`，没有导入 `./styles/index.css`。
- 当前 `frontend/dist/index.html` 也没有任何 CSS 资源链接。
- 用户截图中学生端和教师端都是黑字白底、导航链接默认蓝紫色、按钮默认系统样式，和“CSS 未加载”高度一致。

因此，第 0 阶段不是“美化”，而是恢复样式生效。

## 1. 当前系统阶段判断

### 1.1 产品阶段

当前系统不是早期空壳，而是一个已经进入 `v1.15-stable` 的护理病史采集训练系统，已具备较完整的业务闭环：

- 学生登录、选择病例、开始训练、与虚拟患者对话。
- SSE 流式对话，首字响应体验已优化。
- 训练倒计时、自动结束、自动评分。
- 19 项评分标准，LLM 按 57 分制打分，后端转换为 100 分制展示。
- 评分结果支持 evidence 和 reason，教师可复核。
- 教师端可查看训练记录、管理用户、管理病例、查看 LLM 调用监控。
- 后端已有速率限制、审计日志、健康检查、数据库备份、CSV 流式导出。
- 前端已有 Toast、ConfirmDialog、ErrorBoundary、AppShell、tokens.css 和若干基础 UI 组件。

这说明系统功能已较成熟，下一阶段重点应放在：

1. 恢复并验证样式系统真正生效。
2. 统一学生端和教师端信息架构。
3. 拆分管理后台大组件。
4. 补齐商业级页面状态。
5. 做桌面、平板、手机全尺寸视觉验收。

### 1.2 截图暴露的问题

用户提供的两张截图分别是学生端工作台和教师端仪表盘。截图中可见：

- 页面几乎完全是浏览器默认排版。
- 顶部系统标题、导航链接、用户信息、按钮都没有产品级容器。
- lucide 图标显示出来了，但尺寸和布局没有被 CSS 管控。
- 学生端模块纵向堆叠，缺少卡片、栅格和视觉层级。
- 教师端统计项、快捷操作、图表和表格区域没有明确边界。
- 右侧大面积空白，内容没有进入合理工作台宽度。
- 表格和图表缺少容器、标题区、筛选区和响应式处理。

这些问题优先级如下：

1. 最高优先级：CSS 入口缺失导致所有已写样式不生效。
2. 高优先级：恢复样式后检查 AppShell、Dashboard、Admin 是否符合商业级信息架构。
3. 中优先级：继续拆组件、优化布局、统一控件。
4. 中低优先级：细节动画、图表微交互、视觉质感提升。

## 2. 第 0 阶段：先恢复 CSS 生效

### 2.1 要修改的文件

后续 AI 应首先修改：

```text
frontend/src/main.jsx
```

建议改成：

```jsx
import React from "react";
import ReactDOM from "react-dom/client";
import App from "./App";
import "./styles/index.css";

ReactDOM.createRoot(document.getElementById("root")).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);
```

注意：

- 只加 `import "./styles/index.css";`。
- 不要先改 `dist/`。
- 不要把 CSS link 手写到 `index.html`。
- 让 Vite 构建流程自动把 CSS 打包进 `dist`。

### 2.2 立刻验证

进入前端目录：

```powershell
cd D:\codex\新版\新版\使用版\1.6version\frontend
npm.cmd run build
```

然后检查：

```powershell
Get-Content -Raw .\dist\index.html
```

验收点：

- `dist/index.html` 中应出现类似 `assets/index-xxxx.css` 的 stylesheet 链接。
- `dist/assets/` 下应出现 CSS 文件。
- 启动系统后，截图不应再是浏览器默认样式。

### 2.3 为什么必须先做这一步

当前已有大量样式类：

- `.app-shell`
- `.sidebar`
- `.main-content`
- `.hub-status-bar`
- `.hub-card`
- `.stats-grid`
- `.stat-card`
- `.data-table`
- `.training-shell`
- `.training-topbar`
- `.training-conversation`
- `.score-overlay`
- `.score-modal`

如果 CSS 没有加载，这些类全部失效。后续 AI 如果忽略这一点，会误以为系统从未做过 UI，从而进行不必要的大重写，风险很高。

## 4. 商业级 UI 总方向

### 4.1 产品定位

这是一个护理教育训练工作台，不是营销官网，也不是医院 HIS 系统。视觉方向应该是：

- 专业可信。
- 清爽克制。
- 信息密度适中。
- 学生端强调训练行动和反馈成长。
- 教师端强调管理效率、数据扫视和风险识别。
- 医疗感可以有，但不要冷冰冰。
- 不要做大面积插画、夸张渐变、漂浮装饰或营销 Hero。

### 4.2 色彩策略

沿用当前 `tokens.css`：

- 主色：医疗蓝，负责主按钮、选中态、链接、关键行动。
- 辅助色：青绿色，负责临床/健康/完成感。
- 成功色：绿色，负责优势、完成、通过。
- 警示色：琥珀色，负责待改善、时间提醒、低覆盖率。
- 危险色：红色，负责删除、失败、超时、漏问严重项。
- 大面积背景：浅灰 `#f5f6f8` 或 `--bg-page`。
- 内容表面：白色卡片或表格容器。

不要让页面变成只有蓝色深浅变化的一种色系。

### 4.3 字体和密度

建议继续使用：

```css
font-family: Inter, "PingFang SC", "Microsoft YaHei", system-ui, sans-serif;
```

桌面端密度建议：

- AppShell 侧栏宽度：200 到 220px。
- 顶部局部工具栏高度：56 到 64px。
- 表格行高：44 到 52px。
- 表单控件高度：38 到 44px。
- 主要卡片圆角：8 到 12px。
- 统计卡片 padding：16 到 20px。
- 页面左右 padding：24 到 36px。

商业后台不适合过大的圆角、过厚阴影、过宽留白。

### 4.4 图标策略

继续使用 `lucide-react`。要求：

- 导航图标统一 16px。
- 主卡片图标统一 22 到 28px。
- 表格操作图标统一 14 到 16px。
- 图标不要单独当内容，必须配合文本或 tooltip。
- 不使用 emoji 作为正式 UI 图标。

- 视觉规则难复用。
- hover/focus 状态分散。
- 响应式难维护。
- 组件文件过长，不适合后续继续改。

改造策略：

- 第一轮不要一次清空所有 inline style。
- 优先把重复出现的布局样式抽成组件。
- 单次 PR 或单次 AI 修改只处理一个页面或一类组件。
- 完成一个页面后立即截图验收。

## 6. 学生端优化方案

### 6.1 学生端信息架构

学生端核心目标不是“看很多数据”，而是：

1. 快速开始或继续训练。
2. 清楚知道自己训练进度。
3. 看懂上次反馈并知道下次怎么改。
4. 可随时查护理问题。

推荐学生端导航：

- 首页/工作台：`/home`
- 病例训练：`/cases`
- 训练记录：`/history`
- 护理问答：`/qa`
- 训练统计：`/stats`

移动端底部导航建议保留 3 到 4 项：

- 工作台
- 病例
- 记录
- 问答

统计页可以从工作台或记录页进入，不一定放进移动底栏。

### 6.2 学生 Dashboard `/home`

当前文件：

```text
frontend/src/pages/DashboardHome.jsx
```

当前状态：

- 角色分流已存在。
- 学生端有训练次数、已完成、累计分钟、最新得分。
- 有四张功能卡片。

优化目标：

- 从“功能入口集合”升级为“训练工作台”。
- 页面第一眼就能知道下一步该做什么。
- 继续训练优先级高于新训练。
- 最新反馈要有教学价值，不只是分数。

推荐桌面布局：

```text
顶部 PageHeader：
  左侧：欢迎语 + 训练状态摘要
  右侧：主要操作按钮（继续训练/开始训练）

主内容两列：
  左列 65%：
    当前训练行动卡
    推荐病例/最近病例
    最新反馈摘要
  右列 35%：
    本周训练统计
    快速问答
    能力薄弱项

下方：
  最近训练记录 5 条
```

关键改造点：

- 如果存在 `in_progress` 记录，主按钮显示“继续训练”，并直接进入 `/training/:recordId`。
- 如果没有进行中训练，主按钮显示“选择病例开始训练”。
- 最新反馈摘要不要只显示 `42 分`，应显示：
  - 总分 / 满分。
  - 沟通技能得分。
  - 病史采集得分。
  - 1 条表现较好。
  - 1 条需要改善。
  - 查看完整报告按钮。
- 病例库不要只显示数量，要显示最近推荐的 2 到 3 个病例。
- 快速问答作为辅助功能，不抢主行动。

必须处理状态：

- 病例列表加载中。
- 病例列表为空。
- 训练记录加载中。
- 无训练记录。
- 最新评分加载失败。
- 有进行中训练。

### 6.3 病例选择页 `/cases`

当前文件：

```text
frontend/src/pages/CaseSelect.jsx
```

优化目标：

- 做成真正的病例库。
- 支持学生快速理解病例训练难度，但不泄露诊断。
- 不要一屏只有巨大卡片。

建议结构：

```text
PageHeader：
  标题：病例训练
  副标题：选择虚拟患者开始护理病史采集练习

工具栏：
  搜索框
  难度筛选
  学科/系统筛选（如果后端没有字段，先不显示）

病例列表：
  卡片或紧凑列表
  每项包含：病例名、患者年龄性别、主诉、训练时限、难度、开始训练按钮
```

注意：

- 学生端不要显示诊断。
- `hidden_info` 不显示。
- `required_inquiries` 不直接显示完整答案，只可显示“本病例将训练若干问诊要点”。
- 移动端使用纵向列表。

### 6.4 训练页 `/training/:recordId`

当前文件：

```text
frontend/src/pages/ChatTraining.jsx
```

当前已有优点：

- 独立全屏训练布局。
- 顶部患者信息、倒计时、结束训练按钮。
- SSE 流式回复。
- 语音输入和朗读。
- 采集进度侧栏。

商业级优化目标：

- 像一个真实的护理问诊训练舱。
- 学生聚焦对话，不被过多后台信息干扰。
- 采集进度只提示框架，不泄露答案。

推荐布局：

```text
顶部栏：
  返回
  患者头像/姓名
  病例主诉
  训练倒计时
  采集进度按钮
  结束训练

中间：
  对话流，最大宽度 760 到 840px
  患者消息左侧，学生消息右侧
  系统提示居中

右侧或抽屉：
  采集进度
  护理问诊框架提示
  结束前检查

底部：
  输入框
  语音按钮
  发送按钮
```

改造建议：

- 把 `InquirySidebar` 从 `ChatTraining.jsx` 内部分离到 `components/training/InquirySidebar.jsx`。
- 把顶部栏分离为 `TrainingTopbar.jsx`。
- 把消息气泡分离为 `MessageBubble.jsx`。
- 把输入栏分离为 `TrainingInputBar.jsx`。
- 桌面端采集进度可以是右侧抽屉，移动端必须是底部抽屉或全屏 sheet。
- 输入栏固定底部时要确认不会遮挡最后一条消息。
- 训练时间少于 2 分钟时，倒计时红色，但不要整页恐慌式变红。

必须验收：

- 长患者回复不溢出。
- 语音按钮在不支持浏览器时隐藏或有合理提示。
- SSE 流式状态下发送按钮禁用。
- 结束训练后评分中状态明确。
- 评分失败时能提示去记录页重试。

### 6.5 评分弹窗和记录详情

当前文件：

```text
frontend/src/components/ScoreCard.jsx
frontend/src/pages/RecordDetail.jsx
```

优化目标：

- 从“分数弹窗”升级为“教学反馈报告”。

评分弹窗用于训练结束即时反馈，应该简洁：

- 总分。
- 等级标签。
- 两个维度得分。
- 表现较好 1 到 2 条。
- 需要改善 1 到 2 条。
- 漏问内容摘要。
- 按钮：查看完整报告、再练一次、关闭。

记录详情页用于完整报告，应该详细：

- 训练元信息。
- 对话回放。
- 评分总览。
- 19 项评分可展开。
- evidence 和 reason。
- 教师复核区。
- 导出报告入口。

不要把所有内容都塞进训练结束弹窗，否则学生会被信息淹没。

### 6.6 护理问答 `/qa`

当前文件：

```text
frontend/src/pages/QA.jsx
```

优化目标：

- 像轻量 AI 导师，而不是普通聊天框。

建议：

- 顶部显示推荐问题。
- 支持从 Dashboard 的 `q` 参数自动带入问题。
- 回答区域使用白底卡片，包含复制按钮。
- 支持继续追问。
- 加载状态明确显示“护理导师正在组织答案”。
- 失败状态保留用户问题，允许重试。

## 7. 教师端优化方案

### 7.1 教师端信息架构

教师端不能只是学生端附属功能。教师的核心任务是：

1. 看班级训练总体情况。
2. 找到需要关注的学生。
3. 查看训练记录和评分报告。
4. 管理学生账号。
5. 管理病例。
6. 观察 LLM 调用稳定性和成本。

推荐教师端导航：

- 教学仪表盘：`/home`
- 训练记录：`/history` 或 `/admin?tab=records`
- 学生管理：`/admin?tab=users`
- 病例管理：`/admin?tab=cases`
- 调用监控：`/admin?tab=monitor`
- 训练统计：`/stats`

### 7.2 教师 Dashboard `/home`

当前问题：

- `DashboardHome.jsx` 同时承担学生端和教师端。
- 教师首页虽然有统计卡片、快捷操作、趋势图、最近训练动态，但信息层级还像功能堆叠。

建议新增：

```text
frontend/src/pages/TeacherDashboard.jsx
```

或先在 `DashboardHome.jsx` 中拆出：

```text
frontend/src/components/teacher/TeacherDashboard.jsx
frontend/src/components/student/StudentDashboard.jsx
```

教师 Dashboard 推荐结构：

```text
顶部：
  今日教学概览
  右侧：导出、进入管理后台

第一行统计：
  学生总数
  今日训练次数
  已完成训练
  平均得分
  平均训练时长

第二行：
  训练投入趋势图
  得分分布/能力薄弱项

第三行：
  需要关注学生
  最新训练动态

底部：
  快捷入口：用户管理、病例管理、调用监控、备份
```

商业级关键点：

- 统计卡片要有同比或说明，否则只是数字。
- “需要关注学生”比“快捷操作”更有教学价值。
- 最近训练动态表格要可以直接进入报告。
- 平均分使用颜色标签，不要只有裸数字。

### 7.3 管理后台 `/admin`

当前文件：

```text
frontend/src/pages/Admin.jsx
```

当前问题：

- 单文件非常大。
- 包含记录、用户、病例、批量导入、调用监控、多个弹窗。
- 大量 inline style。
- 后续继续叠功能会越来越难维护。

拆分建议：

```text
frontend/src/pages/Admin.jsx
frontend/src/components/teacher/AdminTabs.jsx
frontend/src/components/teacher/RecordsTab.jsx
frontend/src/components/teacher/UsersTab.jsx
frontend/src/components/teacher/CasesTab.jsx
frontend/src/components/teacher/MonitorTab.jsx
frontend/src/components/teacher/UserEditorModal.jsx
frontend/src/components/teacher/BatchImportModal.jsx
frontend/src/components/teacher/CaseEditorModal.jsx
frontend/src/components/teacher/MonitorStatsPanel.jsx
frontend/src/components/teacher/MonitorLogsTable.jsx
```

拆分原则：

- `Admin.jsx` 只保留 activeTab、权限入口和 tab 容器。
- 每个 Tab 自己管理自己的加载、过滤和数据。
- 表格统一使用 `Table.jsx`。
- Modal 统一使用 `Modal.jsx`。
- 表单统一使用 `FormField.jsx`。

### 7.4 训练记录表

当前已有：

- 学生姓名过滤。
- 病例过滤。
- 状态过滤。
- 日期范围。
- 导出 CSV。
- 删除记录。

商业级补强：

- 显示总条数。
- 分页按钮。
- 加载骨架屏。
- 空状态显示“当前筛选条件无记录”，并提供清除筛选。
- 得分列用颜色标签：
  - 85 到 100：绿色。
  - 70 到 84：蓝色或青色。
  - 60 到 69：琥珀色。
  - 60 以下：红色。
- 状态列用统一 Badge。
- 操作列固定宽度，包含“查看报告”“删除”。
- 小屏时改为横向滚动，不要挤压文字。

### 7.5 用户管理

当前已有：

- 注册用户。
- 编辑用户。
- 删除用户。
- 批量导入。

商业级补强：

- 顶部工具栏：搜索、角色筛选、新增用户、批量导入。
- 用户表格：用户名、姓名、角色、学号、训练次数、最近训练时间、操作。
- 新增用户使用 Modal 或 Drawer，不要占据页面大块空间。
- 批量导入预览必须继续保持密码脱敏。
- 删除用户必须保留 ConfirmDialog。
- 自己账号不可删除的提示要更清晰。

### 7.6 病例管理

当前已有：

- 病例列表。
- 添加/编辑病例。
- JSON 导入。
- 有训练记录时禁止删除。

商业级补强：

- 病例列表卡片或表格要显示：名称、患者、主诉、难度、时限、训练次数、最近编辑时间、状态。
- 新建病例应改成分步骤表单：
  1. 基础信息。
  2. 患者资料。
  3. 病史资料。
  4. 隐藏信息。
  5. 必问清单。
  6. 评分标准预览。
- JSON 高级编辑不要默认展开。
- 保存前做前端校验：名称、主诉、开场白、患者姓名、时限。
- 学生端不能泄露诊断，教师端可以管理完整信息。

### 7.7 调用监控

当前已有 LLM 调用统计和日志。

商业级补强：

- 今日和 7 日统计卡片要统一 `StatCard`。
- 成功率、平均延迟、费用要有颜色规则。
- 日志表格支持用途、状态、日期筛选。
- 错误日志应支持展开查看错误摘要。
- 费用字段注意小数位统一。
- 监控页面属于教师/管理员工具，视觉密度可以比学生端更高。

## 8. AppShell 和响应式优化

### 8.1 桌面端 AppShell

现有：

```text
frontend/src/components/AppShell.jsx
```

建议优化：

- 侧栏宽度从 200px 可调整为 216px。
- 侧栏底部用户区固定，信息不挤压。
- 当前导航项 active 态明显但克制。

  - 数据后台类页面可以撑满。
  - 学生工作台建议 `max-width: 1280px`。

- 或改成顶部栏加抽屉菜单。
- 表格区域允许横向滚动。

- 页面 padding 缩小到 14 到 16px。
- 学生 Dashboard 改为单列。
- 教师表格改成列表卡片或横向滚动表格。
- 训练页输入栏固定底部且不遮挡消息。
- 采集进度用底部抽屉，不用右侧固定抽屉。

## 11. 浏览器视觉验收清单

每次 UI 改完后必须回答以下问题：

1. 页面是否还有浏览器默认按钮、默认链接、默认表单样式。
2. 页面第一屏是否有明确主行动。
3. 学生端是否能一眼知道如何继续训练或开始训练。
4. 教师端是否能一眼看到班级训练概况。

6. 图表是否有标题、说明和容器。
7. 卡片是否过多嵌套。
8. 按钮文字是否拥挤或换行异常。
9. 导航 active 态是否清晰。
10. 图标大小是否统一。
11. 空状态是否有下一步行动。
12. 加载状态是否不会白屏。
13. 错误状态是否能重试。
14. 手机端是否出现横向撑爆。
15. 训练页底部输入栏是否遮挡消息。
16. 采集进度是否泄露答案。
17. 教师端病例管理是否可能泄露到学生端。
18. 评分报告是否能读懂，不只是显示分数。
19. 删除、结束训练等危险操作是否有二次确认。
20. `npm.cmd run build` 是否通过。

## 12. 不要做的事

- 不要直接改 `frontend/dist/`，它是构建产物。
- 不要在没恢复 CSS 的情况下重写所有页面。
- 不要引入大型 UI 组件库，除非用户明确接受依赖和风格迁移成本。
- 不要把学生端做成营销首页。
- 不要在学生端显示诊断、隐藏信息或评分答案。
- 不要把所有页面塞进一个巨大的组件。
- 不要继续在 `Admin.jsx` 里无限追加功能。
- 不要用 emoji 代替正式图标。
- 不要只改颜色和阴影就声称商业级完成。
- 不要忽视手机端。
- 不要忽视浏览器截图验收。
- 不要删除现有安全、评分、复核、病例管理能力。

## 14. 商业级完成标准

可以认为学生端和教师端达到阶段性商业级，必须同时满足：

1. CSS 正常加载，构建产物包含 CSS。
2. 登录、学生端、教师端、训练页都有统一视觉语言。
3. 学生工作台主行动明确。
4. 教师工作台不是学生端改文案，而是真正教学管理视角。
5. 管理后台 Tab 清晰，表格可扫视。
6. 病例管理表单可维护，不是超长混乱表单。
7. 训练页沉浸、稳定、不会遮挡消息。
8. 评分报告有教学反馈价值。
9. 空、加载、失败、无权限、评分中、评分失败状态都有界面。
10. 移动端可用，不只是隐藏侧栏。
11. 所有主流程截图没有明显默认浏览器样式。
12. 所有主流程无文字溢出、遮挡、布局错位。
13. `npm.cmd run lint` 通过。
14. `npm.cmd run build` 通过。
15. `npm.cmd run test` 通过。
16. 后端 compileall 通过。
17. 后续 AI 能在拆分后的组件结构中继续添加功能，而不是继续堆大文件。
